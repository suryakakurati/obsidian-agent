# Benchmarks — Version 1 (SQL Batching Only)

---

## Test 1 — Real Vault (78 notes)

| Batch Checkpoint | Avg / Note | Avg / Embed |
| ---------------- | ---------- | ----------- |
| 30 notes         | 0.0325s    | 0.0322s     |
| 40 notes         | 0.0308s    | 0.0305s     |
| 50 notes         | 0.0330s    | 0.0326s     |
| 60 notes         | 0.0338s    | 0.0334s     |
| 70 notes         | 0.0338s    | 0.0333s     |

| Metric             | Value                                      |
| ------------------ | ------------------------------------------ |
| Batch size         | 50                                         |
| Total index time   | 2.93s                                      |
| Throughput         | 95,760 notes / hour                        |
| Update time        | 0.49s                                      |
| Search latency     | Near-instant (no formal measurement taken) |
| CPU temp           | Not measured                               |
| Similarity quality | Satisfactory                               |

**Observation:** Embedding time accounts for nearly all per-note latency, supporting the hypothesis that embeddings are the primary bottleneck.

---

## Test 2 — Wikipedia Pages (19,513 notes, capped at 500 words)

| Batch Checkpoint | Avg / Note | Avg / Embed |
| ---------------- | ---------- | ----------- |
| 14,000 notes     | 0.1273s    | 0.1257s     |
| 15,000 notes     | 0.1261s    | 0.1245s     |
| 16,000 notes     | 0.1253s    | 0.1236s     |
| 17,000 notes     | 0.1248s    | 0.1232s     |
| 18,000 notes     | 0.1245s    | 0.1229s     |
| 19,000 notes     | 0.1247s    | 0.1230s     |

| Metric             | Value                                      |
| ------------------ | ------------------------------------------ |
| Batch size         | 100                                        |
| Total index time   | 2,433.27s (~40.6 min)                      |
| Throughput         | 28,872 notes / hour                        |
| Update time        | 0.95s                                      |
| Search latency     | Near-instant (no formal measurement taken) |
| CPU temp (peak)    | 169°F                                      |
| Similarity quality | Satisfactory                               |

**Observations:** Longer notes produce meaningfully higher embedding latency. GPU usage was heavy throughout the run. Per-note time stabilised after ~14,000 notes, suggesting consistent throughput at scale. CPU temperature peaked at 169°F, indicating this is a thermally significant workload at scale. This test (19,513 Wikipedia notes, 500-word cap, batch size 100) is the standard benchmark for all future versions.

---

## Comparison

> Test 1 used batch size 50. Test 2 used batch size 100. Keep this in mind when comparing throughput

| Metric          | Test 1 (78 notes) | Test 2 (19,513 notes) |
| --------------- | ----------------- | --------------------- |
| Batch size      | 50                | 100                   |
| Avg / note      | ~0.033s           | ~0.125s               |
| Avg / embed     | ~0.033s           | ~0.124s               |
| Throughput      | 95,760 / hr       | 28,872 / hr           |
| Update time     | 0.49s             | 0.95s                 |
| Search latency  | Near-instant      | Near-instant          |
| CPU temp (peak) | Not measured      | 169°F                 |

Embed time as a share of total per-note time: **~99%** in both tests.
